# 更新业务流程知识库

将新需求或变更需求同步到 `module_knowledge/`，保持知识库与系统现状一致。做到**只改该改的、不碰无关的、改前必确认**。

## 何时使用

- 用户输入新的需求文档（PRD、需求描述、变更说明）
- 用户要求「更新知识库」「同步流程」「新需求入库」
- 用户输入一个已有流程文件路径，要求修改

## 基本原则

1. **改前必确认**：输出变更清单等待用户确认，未收到确认前不写入任何文件
2. **只改列明内容**：变更清单未列出的步骤和描述一律原样保留
3. **表现层不处理**：UI文案、样式、颜色、图标等变化不影响流程，不处理
4. **找不到默认是新增**：`module_knowledge/` 中无匹配文件时，默认是新增内容，请你以该需求文档为核心，增加一个全新的流程知识。
5. **废弃直接删除**：流程步骤废弃时直接从文件中删除对应行，不保留标记
6. **格式严格对齐**：新增或更新内容必须与现有文件格式完全一致，包括编号方式、缩进层级、描述风格

---

## 执行步骤

### 第一步：读取现有知识库

列出 `knowledge/module_knowledge/` 目录下所有 `.md` 文件，理解当前已有的模块信息。

---

### 第二步：分析输入需求

逐段阅读需求文档，提取以下信息：

**A. 识别功能模块**
- 涉及哪些功能模块？
- 是现有模块的变更，还是全新模块？

**B. 识别流程变化**

对每个涉及的功能模块，逐一判断：

| 判断项 | 处理方式 |
|--------|----------|
| 全新的功能模块 | 新增文件 |
| 新增操作步骤 | 在对应文件增加操作，并更新步骤编号 |
| 减少操作步骤 | 删除该操作步骤，并同步修改步骤编号 |
| 步骤描述变更 | 更新对应步骤的描述文字 |

---

### 第三步：给出匹配结果并要求用户确认

按照以下结构输出确认信息

```
═══════════════════════════════════════════

我已经整理完所有信息，以下是接下来需要新增或修改的文件信息：

1. 新增文件「xxxx-xxxx-xxxx.md」并更新 modules.md
modules.md 与 module_knowledge 中均无该模块信息，新增新的模块。

2. 修改文件「xxxx-xxxx-xxxx.md」
将 xxxxx 修改为 xxxxx
在第x步，增加新的流程：xxxxxx
删除第x步：xxxxxxx

═══════════════════════════════════════════
确认后回复「确认新增」继续。
═══════════════════════════════════════════
```

收到「确认新增」后进入第四步。
如本次需求无新增或修改的流程，不输出任何结果，直接进入第四步。

---


### 第四步：执行新增和修改文件

首先，更新 `knowledge/modules.md` 中的模块信息，将新的模块加入到文件中，注意保持原有的模块结构风格，遵循大于自由。

其次，对变更清单中每一条「新增文件」，严格参照以下格式创建文件：

```markdown
已有的[模块名称]流程

1. [第一步描述]
2. [第二步描述]
   2.1 [条件或子步骤描述]
   2.2 [条件或子步骤描述]
3. [第三步描述]
...
```

**文件命名规则**：按模块层级用 `-` 连接，与现有文件命名风格一致。例：`业务参数设置-风控规则-黑名单配置.md`

**流程分类规范**：对于某一个流程，只需要关注新增即可，不需要修改、展示和删除的相关流程。我们的核心是让AI理解先后次序。

**步骤编写规范**：遵循从简原则，只写操作信息，不需要任何额外的规则，如「2. 选择书籍分类」、「3. 输入单车编号」、「8. 上传图片」。均以{编号} + {前一步的条件说明} + {动作} + {字段}的形式输出，允许有根据实际情况小幅变动，但不应脱离基本范式。

最后，请参考以下逻辑，更新已有的模块流程信息

1. 读取原文件全文
2. **仅修改变更清单中明确列出的步骤**，其余内容原样保留，包括文字措辞、标点、缩进
3. 新增步骤时，编号顺延现有编号，缩进层级与同级步骤一致
4. 废弃步骤时，直接删除该步骤信息，并更新步骤编号
5. 修改步骤时，直接修改对应内容为新内容即可。

---

所有文件操作完成后输出：

```
【知识库当前状态】

═══════════════════════════════════════════
```
